# Journal Notes (2026-05-03)

## Repository Safety Cleanup

Today was not about new analysis. It was about making the repository safe to
work in again.

The main problem was that Codex Desktop became dangerous once large generated
and local-only files were still living inside the working tree. That produced
runaway Git pack growth and a repository state that was technically recoverable
but operationally unhealthy.

The useful lesson is straightforward:

- the repo should hold curated source, docs, and small tracked resources
- large derived outputs should live in local data storage
- exploratory agent artifacts should live under `scratch/`

## What Changed

- `.git` was brought back to a sane size after cleanup
- `outputs/` was moved out of the repo to local data storage
- `legacy_resultCellSingles.mat` was moved out of the repo to local data
  storage
- `AGENTS.md` now defines the repository safety rules and maintenance workflow
- `scratch/` is now the explicit required home for exploratory agent-generated
  files

## Why This Matters

This is not only about disk usage.

It is about keeping the project legible to both humans and tools.

Once temporary figures, MAT artifacts, caches, and local environments start
living too close to tracked source, the repo stops being a clean statement of
the project and starts becoming an accidental warehouse. That makes Git
operations less predictable and makes agent tooling more brittle.

## Small Validation After Cleanup

Codex Desktop passed a small scratch-workflow test after the cleanup.

That matters because it shows the current rules are not merely prohibitions.
They also define a workable path:

- exploratory work can still happen
- it just has to happen under `scratch/`
- and it has to be easy to inspect, keep, promote, or delete intentionally

## Practical Rule To Keep

If the output is exploratory, intermediate, generated, or local-only, it should
start in `scratch/` or local data storage, not in tracked project folders.

## Frequency-Space Follow-Up

The next issue was not a new event threshold. It was a rhythmic baseline
oscillation that kept showing up in the envelope and was visible enough that
it could distort the stable-band estimate itself.

The user explicitly noticed that oscillation in the plots and asked about it,
which changed the direction of the analysis. That pushed the work into
frequency space instead of more detector tuning.

What we learned from the raw ACC spectra is that the structure is not purely
an envelope artifact:

- the raw signal already has clear periodic content
- `sub4` looks different from the other subjects
- some windows are much more structured than others
- longer rolling-SD windows suppress the peaky behavior in several cases, but
  not uniformly

So the current status is:

- the oscillation should not drive event detection
- but it should be treated as real signal until proven otherwise
- any suppression method should be evaluated as a preprocessing choice, not
  silently baked into the interpretation

The likely next step is a targeted suppression comparison:

- current envelope
- slower envelope
- raw-signal band-stop / notch candidate around the carrier

The raw-spectrum summary also argues against a single fixed carrier:

- `sub1` tends to peak around `0.5-0.7 Hz`
- `sub3` tends to peak around `~1.0-1.1 Hz`
- `sub4` shifts between roughly `0.6 Hz` and `1.2 Hz`
- `sub2` is different enough that it may not be carrier-dominated in the same
  way

So the stop-band question is no longer "what is the one tone?" but rather:

- what low-frequency neighborhood is consistently nuisance-like enough to
  suppress without erasing the primitive departures we actually care about?

The first direct high-pass preview was informative in a negative sense:

- applying a `1 Hz` zero-phase high-pass before the envelope did not cleanly
  sparsify the detector
- it sharpened the traces, but candidate counts remained high rather than
  collapsing to a small set of primitive events
- the high-pass run produced `755` burst rows and much shorter return times
  than the current non-high-pass envelope runs

So the high-pass idea is still worth keeping as a comparison condition, but
it is not the final solution by itself.

The user then explicitly paused the filtering branch.

The next question should therefore stop asking how to remove the oscillation
and instead ask whether the short primitive events can be recognized by
their shape characteristics.

The shape question to carry forward is:

- are the short events around the putative `~0.5 s` mode morphologically
  consistent across subjects?
- do work and video conditions produce similar or different short-event
  shapes?

That is a better next step than continued threshold tuning because it treats
the current event table as provisional and lets the short-event morphology
tell us whether there is a stable primitive class.

The important part is to keep the oscillatory content as a signal worth
returning to later, while not letting it define the stable band for primitive
departure extraction.
